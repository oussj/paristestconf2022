Problème :

Pendant 24h, tous les utilisateurs qui se connectaient en prod avaient immédiatement un écran blanc sur la première page : le site n'était pas utilisable.

Impact :

Exemple : écran blanc en prod, 100% des utilisateurs impacté pendant 24h

Code :

Le bug apparait à cette ligne dans cette PR : https://github.com/smallable/help-purchase/pull/47/files#diff-b831890ec49920c78cc9e7cf51339aae549c39ed5bdaf241236ef588a4444c05R91-R101

Le bug est fixé ici : https://github.com/smallable/help-purchase/pull/227/files

Interprétation : le problème est présent dans tous les reduce sur le projet

Standard (Flux standard d'exécution du code) :

Exemple :

Contexte : On calcule le prix total pour l'afficher dans le dashboard commande

→ On map sur toutes les références de la commande

→ On map sur les déclinaisons de chaque référence

→ on les fusionne dans une seule liste

Le problème apparait quand il n'y a aucune déclinaison dans aucune référence

→ on somme tous les prix d'achat de tous les éléments de cette liste en utilisant reduce

ce qui déclenche le reduce d'un tableau vide

Analyse

Pourquoi le développeur a-t-il écrit cette ligne ? de cette manière ?

Nezzar a utilisé reduce parce que c'est la bonne manière de faire la somme d'un tableau. Mais il n'a pas initialisé le reduce car il ne savait pas qu'il fallait le faire pour gérer le cas où la liste était vide.

Question : une formation sur le functional programming serait-elle intéressante pour mieux comprendre le concept de reducer, transducer, accumulator ?

Pourquoi le bug n'a pas été détecté plus tôt ? (mur de qualité)

Car le bug n'est pas arrivé plus tôt.

Car le client n'a pas testé de créer des références sans déclinaison.

Car ce cas n'est pas autorisé du point de vue métier, ni théoriquement d'un point de vue technique (faille dans le système de validation qui permet de le faire

Question : faut-il se donner un standard de systématiquement tester les fonctions helpers ? Les faire en TDD ?

Pourquoi le bug a été aussi impactant ?

exemple : pourquoi ça a pris autant de temps ? parce que personne ne tirait, le client n'avait pas trop insisté

exemple : pourquoi toute la page était pétée ?

Il n'y a aucun error-boundary dans l'app.

Action

Action court-terme (qui corrige le bug en prod ASAP)

Rajouter des initial values sur tous les reduce

Action long-terme (qui empêche que ce bug se reproduise)

  • Avoir une règle eslint qui t'empêche d'utiliser reduce sans initialValue voir ici : https://github.com/eslint/eslint/issues/12868
  • Avoir des error-boundary sur le dashboard de commande pour éviter que l'app soit de nouveau pétée

Résultat attendu + date de check

Toute l'équipe sait qu'il faut mettre des initialValues + aucun reduce sans initialValue
